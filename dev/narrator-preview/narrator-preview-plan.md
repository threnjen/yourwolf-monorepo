# Narrator Preview — Plan

> Live narrator instruction preview in the Role Builder wizard

## Overview

When a user builds a role in the Role Builder, a **live narrator preview panel** shows exactly what the narrator would say during the night phase for that role. The preview persists across all wizard tabs and updates in real time as the user modifies the draft. The text is generated by a **backend endpoint** (`POST /roles/preview-script`) to maintain a single source of truth with the existing `ScriptService`.

## A. Requirements & Traceability

### Acceptance Criteria

| ID  | Criterion |
|-----|-----------|
| AC1 | A narrator preview panel is visible on **all four wizard tabs** (Basic Info, Abilities, Win Conditions, Review) |
| AC2 | The preview shows the **full narrator turn**: wake instruction → ability step instructions (with modifiers) → close eyes |
| AC3 | The preview updates **live** as the user changes the draft (name, `wake_target`, ability steps, parameters, modifiers) |
| AC4 | For roles with `perform_immediately` or `perform_as` steps, the preview shows **multiple wake sections** (e.g., Doppelganger: first wake at order 0, then performs copied role's turn) |
| AC5 | The preview text is generated by a **backend endpoint** (`POST /roles/preview-script`) — single source of truth with `ScriptService` |
| AC6 | The preview displays gracefully when the role has no abilities (non-waking role) or the draft is incomplete (shows placeholder/empty state) |
| AC7 | The preview request is **debounced** (~1 s) to avoid flooding the backend on every keystroke |

### Non-Goals

- Adding instruction templates for the 5 missing ability types (`change_to_team`, `perform_as`, `perform_immediately`, `stop`, `random_num_players`) — separate planned feature for Phase 3 completion
- Audio narration preview (Phase 6)
- Timing/duration display in the preview panel
- Preview for multi-role game scripts (this is single-role preview only)

## B. Correctness & Edge Cases

### Key Workflows

1. User on Basic Info tab → types name "My Seer" → preview shows `"My Seer, wake up."` + close eyes (after debounce)
2. User changes `wake_target` to `team.werewolf` → preview updates to `"Werewolves, wake up and look for other werewolves."`
3. User adds View Card step with `target=player.other` → preview adds `"You may look at one other player's card."`
4. User adds second step with OR modifier → preview shows `"OR You may look at 2 cards from the center."`
5. Doppelganger-style role: `view_card → copy_role → perform_immediately` → preview shows Section 1 (own wake) + Section 2 (placeholder for copied role's turn)

### Edge Cases

| Case | Expected Behavior |
|------|-------------------|
| No name yet (< 2 chars) | Preview shows `"[Role Name], wake up."` or panel is hidden |
| `wake_order` is null (non-waking) | Preview shows "This role does not wake up — no narrator instructions." |
| `wake_order` is 0 (Doppelganger-style) | Valid preview generated normally |
| Empty ability steps | Preview shows wake + close eyes only |
| Missing parameter values on a step | Backend generates best-effort text using parameter defaults |
| Backend unavailable | Preview area shows muted "Preview unavailable" — does NOT block wizard |
| Rapid draft changes | Only the last debounced call's response is applied (stale-request guard) |

### Error Handling Strategy

- Preview API failures are **non-blocking** — the preview area degrades to "Preview unavailable"
- The preview endpoint does **NOT** validate the role (that's `/validate`'s job) — it generates text from whatever is provided
- Stale response guard: use an incrementing request ID (matching the existing validation pattern in `RoleBuilder.tsx`)

## C. Consistency & Architecture Fit

### Existing Patterns Followed

| Pattern | Where It Exists | How We Follow It |
|---------|-----------------|------------------|
| `ScriptService` instruction generation | `app/services/script_service.py` | New `preview_role_script()` method reuses all existing `_*_instruction` methods |
| `POST /roles/validate` endpoint | `app/routers/roles.py` | New `POST /roles/preview-script` follows same pattern: takes `RoleCreate` body, returns computed data |
| Pydantic response schemas | `app/schemas/role.py`, `app/schemas/game.py` | New `NarratorPreviewResponse` schema follows `NightScript` structure |
| `rolesApi` methods | `src/api/roles.ts` | New `previewScript()` method alongside `validate()`, using `draftToPayload()` |
| Debounce + stale-request guard | `src/pages/RoleBuilder.tsx` (validation) | Identical pattern for preview calls |
| Inline styles with theme tokens | All `RoleBuilder/` components | `NarratorPreview` component follows same approach |

### Interfaces / Contracts

**Backend request:** Accepts `RoleCreate` (existing schema) — no new request schema needed.

**Backend response — `NarratorPreviewResponse`:**
```
{
  "actions": [
    {"order": 1, "instruction": "Seer, wake up.", "is_section_header": false},
    {"order": 2, "instruction": "You may look at one other player's card.", "is_section_header": false},
    ...
  ]
}
```

**Frontend type — `NarratorPreviewAction`:**
```
{ order: number; instruction: string; is_section_header: boolean }
```

### Deviations

None. All patterns match existing codebase conventions.

## D. Clean Design & Maintainability

### Design

1. **Backend**: Add `preview_role_script(data: RoleCreate)` to `ScriptService`. It constructs a lightweight stand-in object compatible with `_generate_role_script()` and `_get_wake_instruction()` without persisting anything. One new endpoint. One new response schema.
2. **Frontend**: One new `NarratorPreview` component. Debounced API call in `RoleBuilder.tsx` (parallel to validation). Preview state passed through `Wizard` props and rendered on every tab.

### Complexity Risks

| Risk | Mitigation |
|------|------------|
| `_generate_role_script` expects `Role` ORM object with loaded relationships | Build a simple dataclass/named tuple that satisfies the interface — no ORM hackery |
| Two debounced calls (validation + preview) fire on every change | They are independent and can share the same debounce timer, or fire in parallel — both are lightweight |

### "Keep It Clean" Checklist

- [ ] Reuse `_generate_role_script` and all `_*_instruction` methods — no instruction-text duplication
- [ ] Reuse `draftToPayload()` for the API call — no payload duplication
- [ ] Reuse existing debounce pattern from validation flow
- [ ] No new dependencies or libraries added
- [ ] `NarratorPreview` is a pure presentational component (receives data, renders it)

## E. Completeness: Observability, Security, Operability

### Logging

- `ScriptService` already logs at `debug` level for instruction generation — preview uses the same code paths
- The new endpoint logs at `info` level on entry (role name) to aid debugging

### Security

- The preview endpoint accepts the same `RoleCreate` payload as `/validate` — no new attack surface
- All input validation is handled by the existing Pydantic schema
- No data is persisted

### Operability

- **Deploy**: Standard backend + frontend deploy — no migrations, no config changes
- **Verify**: Hit `POST /roles/preview-script` with a sample payload; confirm preview panel appears in UI
- **Rollback**: Revert backend endpoint + frontend component — no schema migrations to undo
- **Monitor**: Standard API latency metrics; preview failures are non-blocking for the user

### Top 5 High-Value Test Cases

**1. Seer preview — full turn**
- **Given** a draft with name="Seer", `wake_target`="player.self", one `view_card(player.other)` step
- **When** preview endpoint is called
- **Then** response contains: `["Seer, wake up.", "You may look at one other player's card.", "Seer, close your eyes."]`

**2. Werewolf preview — team wake + conditional step**
- **Given** a draft with name="Werewolf", `wake_target`="team.werewolf", step 1 `view_awake(team.werewolf)`, step 2 (IF) `view_card(center.main)`
- **When** preview endpoint is called
- **Then** response contains team wake instruction `"Werewolves, wake up and look for other werewolves."` and conditional step text

**3. Non-waking role — empty preview**
- **Given** a draft with `wake_order`=null (non-waking)
- **When** preview endpoint is called
- **Then** response contains empty actions list

**4. Doppelganger preview — multi-section**
- **Given** a draft with `view_card → copy_role → perform_immediately` steps
- **When** preview endpoint is called
- **Then** response contains two wake sections (first wake + copied role section)

**5. NarratorPreview component — visible on all tabs**
- **Given** the `NarratorPreview` component with a list of actions
- **When** rendered within Wizard on each tab
- **Then** the preview panel is visible and shows all action instruction text
